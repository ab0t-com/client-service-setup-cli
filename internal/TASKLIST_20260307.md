## CRITICAL

- [ ] **Script 04: Org-inherited permissions not resolved for non-admin users**
  - File: `scripts/04-setup-end-users-org.sh`, line 310
  - Current: `POST /permissions/grant?user_id=$ADMIN_USER_ID` — writes `subject=user:{admin_id}`
  - Only admin gets the permissions; new users joining the org get nothing
  - **Root cause is TWO gaps, not one:**
    1. **Grant path**: `POST /permissions/grant` (permissions.py:163) always writes `subject=user:{id}`.
       This is the main grant API used by all services (audit, billing, resource, payment, integration,
       auth-mesh-sdk.js). It dual-writes: flat DDB membership record + Zanzibar `subject=user:{id}`.
       This is NOT legacy — it's the active dual-write migration pattern. It does what it should for
       per-user grants. It just doesn't support `subject=org:{id}` because that use case didn't exist yet.
    2. **Resolve path**: `zanzibar_permission_resolver.py:84` resolves from 3 sources:
       (1) role perms, (2) direct user grants, (3) team membership grants.
       It does NOT have a step (4) for org membership grants.
       Even if you wrote `subject=org:{id}` via the Zanzibar API, the resolver wouldn't pick it up.
       Note: `client.py:258` (the low-level Zanzibar check) DOES support org subjects — but the
       resolver used by `GET /permissions/user` and `require_permissions()` does not call it.
  - **Fix (two parts):**
    a) Add step 4 to `zanzibar_permission_resolver.py`: query org memberships, resolve org-level PERM# records
    b) Either extend `POST /permissions/grant` with optional `subject_type` param, or use
       `POST /zanzibar/permissions/grant` (already supports `subject: "org:{id}"`) in the script
  - **Callers of `POST /permissions/grant` (blast radius if changed):**
    - `auth-mesh-sdk.js` (2 copies: dashboard + hosted_login)
    - `register-service-permissions.sh` (audit, billing, billing/banking, resource, integration, payment)
    - `00_setup_*_admin.sh` (integration, payment)
    - UJ test scripts (payment: 27, 28, 29, 30, 31)
    - All use `?user_id=X` — per-user grants. None attempt org-level grants.
    - Conclusion: adding an optional `subject_type` param is backward-compatible.
  - **Workaround**: Script 04.1 (team-inherited) works today — teams are fully wired end-to-end
  - TODO comment added inline: `scripts/04-setup-end-users-org.sh` line 297

## Zanzibar Migration Status (context for CRITICAL fix)

The system is mid-migration. The dual-write pattern is correct and intentional:

```
WRITE PATH (current — working as designed):
  POST /permissions/grant → flat DDB (USER# membership.permissions[]) + Zanzibar (subject=user:{id})
  POST /teams (create)    → flat DDB (ORG# TEAM#) + Zanzibar (subject=team:{id}) via on_team_created
  POST /teams/membership  → flat DDB + Zanzibar relationship (user→team member) via on_team_member_added

READ PATH (current — working as designed):
  GET /permissions/user → zanzibar_permission_resolver.py (primary) → flat fallback
  Resolver walks: (1) role → (2) direct user PERM# → (3) team membership PERM#
  Missing: (4) org membership PERM# ← this is the gap

LOW-LEVEL ZANZIBAR (separate, not used by main auth flow):
  POST /zanzibar/check         → client.py check() — supports user/team/group/org subjects
  POST /zanzibar/permissions/* → direct Zanzibar read/write, requires zanzibar.admin
```

The `POST /permissions/grant` API is not legacy — it IS the dual-write bridge. It correctly
writes to both stores. It just only supports `subject=user:{id}` because org-level grants
weren't needed until now. The fix is additive, not a migration.
