# Setup CLI Structural Audit

**Date**: 2026-03-07
**Scope**: All scripts (01-05, 04.1), all config files, all credential outputs
**Method**: Code read + dependency trace + end-to-end flow simulation

---

## Verdict

**Script 04.1 (team-inherited): PASS** — would work end-to-end for a new service.

**Script 04 (org-inherited): FAIL** — grants permissions to ONE user (admin),
not to all org members. New users signing up via hosted login get zero service
permissions. The comments claim "org-level permissions inherited by all members"
but the code does `POST /permissions/grant?user_id=$ADMIN_USER_ID` which is a
per-user grant, not an org-level grant.

**Scripts 01, 02, 03, 05: PASS** — no issues found.

---

## Findings

### CRITICAL: Script 04 does not work for end users

**File**: `scripts/04-setup-end-users-org.sh`, lines 297-326

**What the code does** (line 310):
```bash
POST /permissions/grant?user_id=$ADMIN_USER_ID&org_id=$END_USERS_ORG_ID&permission=$PERM
```

This grants each `default_grant` permission to the **admin user only**.
It creates a Zanzibar Permission record: `subject=user:{admin_user_id}`.

**What the comments say** (lines 297-298):
```bash
# Grant default_grant permissions at the org level.
# These become the baseline for all members via Zanbibar inheritance.
```

**What actually happens when a new user signs up**:
1. User registers via hosted login at `/login/{service-id}-users`
2. Auth server creates org membership (role: member)
3. User has role-based permissions only: `api.read`, `api.write`, `org.read`
4. User has ZERO service permissions (e.g., `integration.read.services`)
5. User gets HTTP 403 on all service endpoints

**The output file says** (line 405):
```
Permissions: 16 inherited via org membership
```

This is false. Only the admin has them.

**Impact**: Any service using script 04 will have broken end-user access.
The admin can use the service; nobody else can.

**Fix options**:
- Use script 04.1 instead (team-based, proven working)
- Or: implement org-level permission inheritance in the Zanzibar resolver
  (currently not supported — the flat resolver does not walk org hierarchy
  for permissions, only for relationships)

### HIGH: `permissions.json` has stale `auto_provision` field

**File**: `config/permissions.json`, lines 305-311

```json
"end_users": {
    "auto_provision": true,
    "provision_on": "org_join",
```

The `auto_provision: true` and `provision_on: "org_join"` fields reference
auto-provisioning code that was removed from `auth_service_base.py` (it caused
a permission leak — see COMPLIANCE_NOTE_auto_provisioning_removal.md).

These fields are not consumed by any script or server code. They are
informational, but they are misleading — someone reading this config would
think auto-provisioning is active.

**Fix**: Update to reflect the actual mechanism:
```json
"end_users": {
    "auto_provision": false,
    "default_team_name": "Default Users",
    "provision_via": "default_team",
    "notes": "..."
}
```


### MEDIUM: Scripts 02 and 03 default SERVICE_ID to `integration-service`

**Files**:
- `scripts/02-register-oauth-client.sh`, line 76: `SERVICE_ID="${SERVICE_ID:-integration-service}"`
- `scripts/03-setup-hosted-login.sh`, line 68: `SERVICE_ID="${SERVICE_ID:-integration-service}"`

These hardcode `integration-service` as the default SERVICE_ID. A new service
with `service.id = "billing"` in permissions.json would need to either:
- Set `SERVICE_ID=billing` env var when running 02 and 03
- Or the scripts would look for `credentials/integration-service.json` and fail

Scripts 01, 04, and 04.1 read SERVICE_ID from `permissions.json` correctly.
Scripts 02 and 03 do not read `permissions.json` at all — they use a hardcoded
default.

**Fix**: Scripts 02 and 03 should read SERVICE_ID from permissions.json:
```bash
SERVICE_ID="${SERVICE_ID:-$(jq -r '.service.id' "$PERMISSIONS_FILE")}"
```

### MEDIUM: Hardcoded `integration-service` credential fallback

**Files**:
- `scripts/04-setup-end-users-org.sh`, lines 95-98
- `scripts/04.1-setup-default-team.sh`, lines 109-112

Both have a fallback credential path:
```bash
CREDS_FILE="$SETUP_DIR/credentials/integration-service${ENV_SUFFIX}.json"
```

This is a legacy fallback from when the service was always called
"integration-service". A new service with `service.id = "billing"` would
never hit this fallback (01 writes `credentials/billing.json`), so it's
harmless but confusing.

**Fix**: Remove the `integration-service` fallback or replace with a
generic glob pattern.

---

### LOW: `org-structure.json.example` and `archetypes/` are dead config

**Files**:
- `config/org-structure.json.example`
- `config/archetypes/*.json`

These files are not consumed by any script. They define a `tiers[]` model
with `permission_model`, `capabilities`, `membership` fields — a more
generic version of what scripts 04/04.1 hardcode.

Not a bug — they serve as design documentation and future reference. But
a user might think they need to configure `org-structure.json` before
running the scripts.

**Recommendation**: Add a note in the README or in the file itself:
"This file is a design reference. Scripts 04/04.1 do not read this file."

---

## Data Flow Verification

### Script 01 → credentials/{service-id}.json

```
permissions.json
  ├─ service.id → determines credential filename
  ├─ registration.{service,actions,resources} → POST /permissions/registry/register
  └─ permissions[].implies → POST /permissions/grant (admin only)

Output: credentials/{service-id}.json
  Contains: organization.id, admin.{email,password,user_id}, api_key.{id,key}
```

**Status**: PASS. Reads from config, writes to credentials. No hardcoded values
beyond AUTH_SERVICE_URL default.

### Script 02 → credentials/oauth-client.json

```
oauth-client.json → POST /auth/oauth/register (entire payload)
credentials/{service-id}.json → admin login for auth

Output: credentials/oauth-client{env-suffix}.json
  Contains: client_id, registration_access_token
```

**Status**: PASS (with SERVICE_ID caveat above).

### Script 03 → credentials/hosted-login.json

```
hosted-login.json → PUT /organizations/{org_id}/login-config (entire payload)
credentials/{service-id}.json → admin login for auth

Output: credentials/hosted-login{env-suffix}.json
  Contains: hosted_login_url, verification results
```

**Status**: PASS (with SERVICE_ID caveat above).

### Script 04 → credentials/end-users-org.json

```
permissions.json → default_grant permissions, default role
hosted-login.json → login config (modified: default_role=member)
credentials/{service-id}.json → admin login, service org ID

Output: credentials/end-users-org{env-suffix}.json
  Contains: org_id, permission_model=org-inherited
```

**Status**: FAIL. Grants permissions to admin user only. New users get nothing.

### Script 04.1 → credentials/end-users-org.json

```
permissions.json → default_grant permissions, default_team_name, default role
hosted-login.json → login config (modified: default_role=member, default_team={id})
credentials/{service-id}.json → admin login, service org ID

Output: credentials/end-users-org{env-suffix}.json
  Contains: org_id, default_team_id, permission_model=team-inherited
```

**Status**: PASS. Team created with correct permissions. Login config points to
team. New users auto-join team on registration. Permissions flow via Zanzibar
graph traversal. Proven by UJ-178 (8/8 GREEN) and UJ-179 (8/8 GREEN).

### Script 05 (verify)

```
Reads all config + credential files. Makes read-only API calls.
Reports PASS/WARN/FAIL.
```

**Status**: PASS. Read-only, no side effects.
